## Recommended build order

`A → B → D → C → G → (E, F any time)`

Rationale: A produces correctly-parsed bodies + the `snippet` field everything downstream consumes;
B depends on clean bodies; D reshapes the data layer that C/G read from; C and G share `EmailsPage.tsx`
so they go back-to-back; E and F are independent and can slot in anywhere.

## Cross-plan reconciliation notes (from self-review)

1. **C tokenizer is already fixed.** The audit listed a URL/time tokenizer bug; on inspection
   `searchParser.ts:183,195` already has an operator allowlist, so `https://x` and `12:30` already
   fall through to free text. Bucket C's Task 1 is therefore a **regression-guard test**, not a fix.
   The real bug — `searchParser` never wired into the UI — remains and is fixed in C.

2. **✅ RESOLVED (chosen: reconcile, run D→C).** Bucket D (option A) moves
   `body`/`htmlBody`/attachment `data` out of the in-memory store rows and adds a bounded
   `Email.searchText?` field (first ~2KB of stripped body). Bucket C wires `searchParser.filterEmails`,
   whose `body:` and free-text matching read `email.body` in memory — which is absent post-D.
   **Resolution applied to the plans:** C's new **Task 1.5** changes the two body reads in
   `filterEmails` (searchParser.ts:251 and :283-287) to `email.searchText ?? email.body`. This is
   backward-compatible (falls back to `body` when `searchText` is absent, i.e. if D hasn't landed).
   D owns adding `Email.searchText?` + populating it; D's interim edit of the EmailsPage `.includes`
   list-search is **superseded** by C's `filterEmails` swap (Task 3) — expected, not a regression.
   Accept that full-text body search is bounded to the truncated `searchText` after D.

3. **D blast radius (already handled in its plan):** removing body from rows also affects body-substring
   reads in `SenderEmailsPage.tsx:102,343` and `EmailDetailPage.tsx:252-278`; D's plan re-sources those
   via `useLazyEmailBody` and a bounded `searchText`. If option (b) above is chosen, those D steps drop too.

4. **⚠️ D → E backup dependency (discovered during D execution, commit e390a23).** After D's table split,
   `backupService` exports only `db.emails` rows — which no longer contain `body`/`htmlBody`/attachment
   `data` (those moved to the new `emailBodies` table) — and it does not touch `emailBodies` at all. So a
   backup created AFTER the v5 migration silently loses all email bodies on restore. **Bucket E's plan
   MUST be extended** to export AND import the `emailBodies` table (and the `searchText` field) so backups
   are complete. Patch Bucket E before executing it. Existing backup tests still pass only because they
   don't assert on body content — add a round-trip body-integrity assertion in E.
